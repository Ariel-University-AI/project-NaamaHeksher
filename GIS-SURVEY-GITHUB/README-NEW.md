## 🔭 סקירה כללית

**GIS — Geographic Intelligence System** היא מערכת מקומית שמטרתה היחידה היא **לחזות ולעקוב אחר רווחיות פרויקטי מדידה**. כל מרכיב במערכת — המפה, מאגר הפרויקטים, חילוץ ה-DWG, התיוג — קיים כדי לשרת את השאלה: *"כמה רווחי יהיה הפרויקט הבא, ועד כמה הפרויקטים הקודמים באמת היו רווחיים?"*

### 🎯 הגדרת רווחיות במערכת

המערכת עובדת עם **מודל רווחיות פשוט ומדיד**:

```
רווח = quoted_price − (actual_hours × hourly_cost)
שולי רווח (%) = רווח / quoted_price × 100
```

זה מה שמאפשר לכל פרויקט שמסתיים להפוך אוטומטית לנקודת נתונים שמזינה את מודל החיזוי.

### שתי פונקציות הליבה

| פונקציה | מתי | מה |
|----------|------|-----|
| 📊 **מעקב רווחיות (Hindsight)** | אחרי שפרויקט נסגר | חישוב רטרואקטיבי של רווח/הפסד לכל פרויקט. איתור פרויקטים מפסידים, זיהוי דפוסים: אילו סוגי מדידה, אזורים, לקוחות הפסידו לנו כסף? |
| 🔮 **חיזוי רווחיות (Foresight)** | כשנכנסת פנייה חדשה | אומדן שעות עבודה צפויות לפרויקט החדש על סמך פרויקטים דומים בעבר, ומכאן — טווח רווחיות לכל מחיר הצעה אפשרי |

המערכת מתוכננת כ**כלי ייעודי וצנוע**: כל פיצ'ר שלא תורם ישירות לאחת משתי הפונקציות הללו — נזרק.

### 🌐 למה שרת מקומי בדפדפן?

| סיבה | הסבר |
|-------|-------|
| 🔒 **פרטיות מלאה** | קבצי המדידה לא עוזבים את המחשב — לא לענן, לא לשירות צד שלישי |
| ⚡ **גישה לכלי דסקטופ** | המרת DWG דורשת תוכנה מקומית (ODA File Converter), והשרת המקומי קורא לה ישירות |
| 🌍 **חוויית Web מלאה** | פיתוח באמצעות טכנולוגיות web סטנדרטיות (HTML/CSS/JS + Leaflet) עם כל היתרונות, בלי תלות באינטרנט |
| 💰 **עלות אפס** | אין שרת בענן, אין מנויים, אין הוצאות חודשיות |
| 🛠️ **פשטות תחזוקה** | רכיב אחד פיזי לתחזוקה — המחשב של מנהל המשרד |

### מה המערכת מאפשרת

כל יכולת ברשימה הזו מסומנת ב-📊 (משרתת מעקב) או 🔮 (משרתת חיזוי) — אם פיצ'ר לא מסומן בלפחות אחד מהם, הוא לא נכלל.

- 📊🔮 **קליטה אוטומטית של DWG** — חילוץ פוליגון תחום המדידה (הבסיס לחישוב גודל וקרבה גאוגרפית, שני המנבאים החזקים ביותר של שעות עבודה)
- 📊🔮 **תיוג מטא-דאטה עסקית** — לקוח, סוג מדידה, גוש/חלקה, **מחיר הצעה, שעות בפועל, עלות שעה** (השדות הקריטיים למודל הרווחיות)
- 🔮 **חיפוש מרחבי** — איתור פרויקטים קודמים באותו אזור כקלט לחיזוי רווחיות לפנייה חדשה
- 📊 **דשבורד רווחיות** — אילו פרויקטים הרוויחו / הפסידו, התפלגות שולי רווח לפי סוג מדידה / לקוח / אזור
- 🔮 **מנוע "פרויקטים דומים"** — k-Nearest Neighbors שמחזיר את N הפרויקטים הקרובים ביותר לפנייה חדשה, ומכאן טווח שעות וטווח רווח צפויים
- 📊🔮 **מפת רווחיות** — Heatmap גאוגרפי שמראה איפה אנחנו מרוויחים ואיפה לא, כדי לכוון החלטות תמחור עתידיות
- 📊 **דוח רווחיות חודשי / שנתי** — סיכום אוטומטי של פרויקטים שנסגרו, רווח כולל, ופרויקטים שדרשו שינוי הצעת מחיר

### ✨ שמירה וטעינת עבודה — Session Management (v2.5)

אחרי שלב 2, כל הנתונים שפורסרו (`_parsed_cache` + `_survey_cache`) חיים רק בזיכרון השרת. כיבוי השרת מאבד את כל העבודה. יכולת ה-session פותרת זאת:

| פעולה | תיאור |
|--------|--------|
| **💾 שמור עבודה** | לחצן מופיע לאחר סיום שלב 2. המשתמש מזין **שם לעבודה** ובוחר **תיקיית שמירה** — נוצר קובץ `<שם>.session` (gzip-compressed JSON) |
| **📂 טען עבודה** | לחצן קבוע בסרגל העליון. פותח חלונית בחירת קובץ `.session`. לאחר טעינה, שם העבודה מוצג בבאדג' סגול ומערכת הייצוא זמינה מיד |
| **ביטול טעינה** | לחיצה על ✕ בבאדג' מסתירה את השם (הנתונים נשארים בזיכרון) |

**מבנה קובץ `.session`** (לאחר פריסת ה-gzip):
```json
{
  "version": "1",
  "saved_at": "2026-05-25T14:30:00",
  "name": "עבודה 12633-RZ",
  "map_count": 24,
  "total_rows": 48520,
  "parsed_layer": "M1200",
  "settings": { "output_dir": "C:\\...", "layer": "M1200" },
  "maps":   { "12633-RZ": [ { "Layer": "M1200", "X": "...", ... }, ... ] },
  "survey": { "12633-RZ": { "count": 12, "map_type": "RZ", "block_names": ["FORM_5"] } }
}
```

> 💡 **גודל קובץ אופייני**: פרויקט עם 20–30 מפות ≈ 5–15MB (לאחר דחיסת gzip). קובץ `.session` שנשמר ב-SURVEYGIS ניתן לטעינה גם ב-SYSTEM ולהיפך (פורמט משותף).

### ✨ יכולות Pipeline שנבנו (v2.1)

#### חילוץ DWG/DXF

| יכולת | מימוש |
|--------|--------|
| זיהוי סוג מפה אוטומטי | `_auto_layer()` — RZ / TZR לפי pattern שם הקובץ |
| חילוץ מטה-דאטה RZ | בלוקי `MAP_TITLE`, `AREA_TBL_ROW`, `C1640` עם ATTRIBs |
| חילוץ מטה-דאטה TZR | בלוקי `TZR_TITLE_1`, `C1643`, `DIVIDE_HEADER` |
| שחזור טקסט עברי | `_fix_hebrew()` — תיקון קידוד ODA על ATTRIBs מושחתים |
| ייצוא SHP עם מטה-דאטה | שדות DBF מלאים עם aliasing ל-10 תווים (`FIELD_ALIASES`) |

#### ✨ ספירת נקודות מדידה (`count_survey_points`)

| סוג מפה | בלוק מדידה | חיפוש |
|---------|------------|--------|
| RZ (רישום זכויות) | `M1502_P` ו-`TRP*` | model space + רקורסיבי |
| TZR (תצ"ר) | `C1670` | רקורסיבי בתוך בלוקי מיכל |

הפונקציה מזהה אוטומטית את סוג המפה, סורקת רקורסיבית את עץ הבלוקים, ומחזירה מונה + סוג מפה. התוצאה מוצגת ב-panel המפה כ-chips צבועים לכל קובץ.

#### API endpoints מרכזיים

| Endpoint | תפקיד |
|----------|--------|
| `GET /api/metadata` | מטה-דאטה שחולצה מ-DXF + **`survey` — ספירת נקודות מדידה** |
| `GET /api/parsed-geojson` | גאומטריות GeoJSON לתצוגת מפה |
| `GET /api/shplist` | רשימת SHP קיימים |
| `GET /api/geojson` | SHP→GeoJSON לדשבורד |
| `GET /api/map-types` | הגדרות MAP_TYPES + MAP_TYPE_RULES |
| `GET /api/browse-file` | פותח חלונית בחירת קובץ `.session` (tkinter) |
| `POST /api/step1` | סריקה + המרת DWG→DXF (SSE streaming) |
| `POST /api/step2` | פרסור DXF לזיכרון (SSE streaming) |
| `POST /api/export` | ייצוא SHP/KML/CSV/GeoJSON |
| `POST /api/save-session` | שמירת `_parsed_cache` + `_survey_cache` לקובץ `.session` (gzip JSON) |
| `POST /api/load-session` | טעינת קובץ `.session` לזיכרון — מחזיר metadata לעדכון ה-UI |

### ✨ פאנל "שלב 4 — נתוני אימון" (v2 — ממומש ב-SURVEYGIS)

מאפשר **התחלת איסוף דאטה היסטורי מיד**, בלי לחכות ל-DB.
הגרסה המשודרגת (`SURVEYGIS/`) מחשבת הכל בצד-שרת ומסנכרנת אוטומטית:

| יכולת | מימוש |
|--------|--------|
| auto-fill מטה-דאטה מה-DXF | `GET /api/training-data` — שולף מ-`_parsed_cache`: client, gush, helka, city, survey_date |
| `area_sqm`, `perimeter_m`, `shape_index` | מחושב ב-Python מהפוליגון הראשי (`_build_training_rows`) |
| `center_lat/lon` | המרת ITM → WGS84 ב-`pyproj` (דיוק מלא) |
| `distance_km` | Haversine ב-Python מול מיקום משרד מה-query params |
| `survey_points` | נשלף ישירות מ-`_survey_cache` — **חדש, לא היה בגרסה קודמת** |
| `official_area_sqm` | שדה `At_AREA` מה-DXF — שטח רשמי לעומת מחושב |
| `map_type` (RZ/TZR) | מ-`_survey_cache`, read-only בטבלה |
| `finish_date` | מ-`F5_FINISH_` / `C43_FINISH` בה-DXF |
| `is_accepted` | ידני — האם ההצעה התקבלה (לעתיד: win-rate model) |
| שמירה אוטומטית | debounce 1.5s → `POST /api/training-save` → `training_data.json` בצד-שרת |
| טעינת היסטוריה בפתיחה | `GET /api/training-load` — נתונים נשמרים בין הפעלות |
| ייצוא CSV | 26 עמודות, BOM לתאימות עברית באקסל |
| **📥 ייבוא CSV** | כפתור מפעיל `<input type="file">` — הקובץ נקרא ב-`FileReader` בצד הלקוח בלבד (ללא בקשת שרת). שדות שלא קיימים בCSV נשארים כפי שהם; שדות קיימים בCSV מחליפים ערכים קיימים (merge לפי `project_id`). שורות ללא `project_id` מתווספות כשורות חדשות. |
| **📐 עדכן חישובים** | מחיל את ה-`hourly_cost` מה-input "עלות לשעה" על **כל** השורות. אם מוגדרים קואורדינטות משרד, מחשב מחדש `distance_km` לכל שורה עם קואורדינטות מרכז (`haversineM`). לאחר מכן מריץ `dtRecomputeFinance` לעדכון `profit` + `profit_margin_pct`. |
| **🗑 נקה הכל** | מציג confirm עם מספר השורות לפני מחיקה. אחרי אישור — מרוקן את `DT_ROWS`, מרנדר טבלה ריקה, ומפעיל שמירה אוטומטית לאיפוס `training_data.json`. |

#### תהליך הזנת נתונים לחיזוי (Zone A)

הפאנל תומך בשתי שיטות קלט חלופיות — שתיהן מחשבות אוטומטית את כל ה-features הגאומטריים:

| שיטה | תיאור | נקודות מדידה |
|------|--------|---------------|
| **✏️ ציור פוליגון על מפה** | כפתור "✏️ צייר פוליגון" מעל מפת Leaflet עם בחירת מפת רקע (Satellite / Topo / OSM / Dark). לחיצה כפולה סוגרת את הפוליגון. | שני צירים: נוסחה + k-NN (ראה להלן) |
| **📁 קובץ DWG** | הזנת נתיב קובץ DWG קיים; השרת מחלץ את שכבת `M1200` (RZ) או `C1602_0` (TZR) | ספירה בפועל מה-DXF (מדויקת) |

לאחר קלט הפוליגון, מוצגת סרגל סטטיסטיקות: שטח ∙ היקף ∙ מדד צורה ∙ מרחק מהמשרד ∙ **📐 נוסחה** ∙ **🤖 ML**.  
המשתמש מוסיף ידנית: **סוג מדידה** (RZ/TZR/other) + **סוג שטח** (פתוח/משולב/בנוי) + **מחיר הצעה** → לחצן "חזה" מופעל.

> **שינוי סוג מדידה או סוג שטח** — מפעיל `prReCalcIfPoly()` שמחשב מחדש את הגאומטריה ושולח שוב ל-`/api/calc-geometry`. אם תוצאת חיזוי כבר מוצגת, היא מתרעננת אוטומטית.

#### שערוך נקודות מדידה — שני צירים

**ציר 1 — נוסחה גולמית** (פועל תמיד, ללא תלות בנתוני אימון):

```
צפיפות_ראשונית = 200 ÷ 500 = 0.4 נק'/מ"ר
נקודות = (D₀ ÷ k) × (1 − e^(−k × שטח)) × מקדם_מפה × מקדם_שטח
```

הצפיפות יורדת אקספוננציאלית ככל שהשטח גדל (k = 5×10⁻⁵), אך **סך הנקודות תמיד עולה עם השטח**.

| פרמטר | ערך | מקדם |
|--------|-----|------|
| סוג מפה | RZ | 1.00 |
| | TZR | 0.50 |
| | אחר | 0.75 |
| סוג שטח | פתוח | 0.25 |
| | משולב *(ברירת מחדל)* | 0.75 |
| | בנוי | 1.00 |

דוגמה — שטח משולב, RZ:

| שטח | נקודות | צפיפות (נק'/מ"ר) |
|-----|--------|-----------------|
| 500 מ"ר | ~110 | 0.29 |
| 2,000 מ"ר | ~428 | 0.21 |
| 10,000 מ"ר | ~1,771 | 0.18 |
| 50,000 מ"ר | ~4,131 | 0.08 |

**ציר 2 — k-NN היסטורי** (פועל כשיש ≥1 פרויקט בנתוני האימון עם `survey_points` מלא):  
מחפש k=5 פרויקטים קרובים ביותר לפי יחס שטח + סוג מפה, ומחזיר ממוצע נקודותיהם.  
**מוצג בלבד כייחוס** — אינו מוזן למודל הרווחיות כי אינו מכיר terrain.

> **מה נכנס למודל**: תמיד `survey_points_raw` (ציר 1, מותאם terrain). ציר 2 מוצג ב-UI להשוואה בלבד.  
> **תכנון עתידי**: השוואת ציר 1 מול ציר 2 כשיצטברו נתונים → כיול מקדמי הנוסחה.

### הפתרון — לולאת רווחיות סגורה

```
פנייה חדשה  →  חיזוי שעות מבוסס פרויקטים דומים  →  המלצת מחיר עם טווח רווח
```

המערכת סוגרת את הלולאה: **כל פרויקט שמסתיים הופך לנקודת נתונים שמשפרת את החיזוי לפרויקט הבא**. ב-1–2 משתמשים (מנהל המשרד), בלי ניהול משתמשים, הרשאות או תהליכי צוות.

### קבצי מדידה נתמכים

| פורמט | סיומת | תיאור | תהליך |
|--------|--------|--------|--------|
| AutoCAD DWG | `.dwg` | פורמט מקורי של AutoCAD / Civil3D | ODA File Converter ממיר ל-DXF → ezdxf מחלץ |
| AutoCAD DXF | `.dxf` | פורמט חליפין | ezdxf קורא ישירות |

> ⚠️ **מודולים נוספים (KML, GPX, LAS, תמונות אווריות) נדחים לשלבים מאוחרים** — לא נכללים ב-MVP. נוספים רק במידה ויש צורך אמיתי שעולה משימוש בפועל.

| שדה | סוג | חובה | תיאור |
|------|-----|------|--------|
| `project_name` | טקסט | ✅ | שם תיאורי לפרויקט |
| `survey_type` | בחירה מרשימה | ✅ | טופוגרפית / לתכנון / לקדסטר *(מנבא חזק של שעות)* |
| 🔮 `site_type` | בחירה | ✅ | תנאי השטח: `open` (פנוי) / `mixed` (מעורב) / `built` (בנוי) — **מקדם קושי בחיזוי** |
| `client_name` | טקסט | ✅ | שם הלקוח / חברה |
| `survey_date` | תאריך | ✅ | תאריך ביצוע המדידה בשטח |
| `gush` | מספר | ⚪ | מספר גוש |
| `helka` | מספר | ⚪ | מספר חלקה |
| `address` | טקסט | ⚪ | כתובת או אזור |
| `city` | טקסט | ⚪ | עיר / יישוב |
| `status` | בחירה | ✅ | בעבודה / הושלם / הוגש / ארכיון |
| 💰 `quoted_price` | מספר (₪) | ✅ עם סגירה | **קריטי לרווחיות** — מחיר ההצעה שאושר ע"י הלקוח |
| 💰 `actual_hours` | מספר | ✅ עם סגירה | **קריטי לרווחיות** — סך שעות העבודה בפרויקט (משרד + שטח) |
| 💰 `hourly_cost` | מספר (₪) | ✅ עם סגירה | **קריטי לרווחיות** — עלות שעת עבודה ממוצעת (יכול להיות ערך ברירת מחדל גלובלי שמתעדכן) |
| 📐 `area_sqm` | מספר | אוטומטי | שטח הפוליגון בממ"ר — מחושב אוטומטית מ-DWG, מנבא חזק של שעות |
| `surveyor_notes` | טקסט חופשי | ⚪ | הערות פנימיות |

> 💡 **שדות מחושבים אוטומטית מ-3 שדות הליבה**:
> - `profit = quoted_price − (actual_hours × hourly_cost)`
> - `profit_margin_pct = profit / quoted_price × 100`
> - `effective_hourly_rate = quoted_price / actual_hours` (כמה למעשה הרווחנו לשעה)

> 💡 **שדות עתידיים** — נשמרים בסכמה מהיום גם אם לא מולאו ב-MVP, כדי לשפר את החיזוי בעתיד: `complexity_score`, `revision_count`, `client_repeat_count`. הם לא חוסמים את שימושיות המערכת — מודל הרווחיות הבסיסי עובד בלעדיהם.

## 🗄️ בסיס נתונים

### טכנולוגיה

**PostgreSQL 16 + PostGIS 3.4** — מקומי, רץ על אותו מחשב כמו השרת.

> 💡 **למה לא SQLite + SpatiaLite?** — SQLite היה מספיק לחלוטין ל-1–2 משתמשים, אבל PostGIS נותן יכולות מרחביות עשירות יותר (בעיקר אינדקסים GIST, ופונקציות ניתוח מתקדמות) שיהיו חשובות בשלב 4 לחיזוי וניתוח. ההתקנה חד-פעמית, התחזוקה אפסית.

### טבלאות עיקריות — פירוט

#### `projects` — פרויקטים

```sql
CREATE TABLE projects (
    id                  SERIAL PRIMARY KEY,
    project_name        VARCHAR(300) NOT NULL,
    client_id           INTEGER REFERENCES clients(id),
    survey_type_id      INTEGER REFERENCES survey_types(id),
    status              VARCHAR(20) DEFAULT 'in_progress'
                        CHECK (status IN ('in_progress','completed','submitted','archived')),
    gush                VARCHAR(10),
    helka               VARCHAR(10),
    address             TEXT,
    city                VARCHAR(100),
    survey_date         DATE,
    site_type           VARCHAR(10) CHECK (site_type IN ('open','mixed','built')),
    -- features מחושבים אוטומטית מהפוליגון (לחיזוי)
    area_sqm                NUMERIC(12,2),
    perimeter_m             NUMERIC(12,2),
    shape_index             NUMERIC(6,3) GENERATED ALWAYS AS
                            (CASE WHEN area_sqm > 0
                                  THEN (perimeter_m * perimeter_m) / (4 * pi() * area_sqm)
                                  ELSE NULL END) STORED,
    distance_from_office_km NUMERIC(6,2),  -- מחושב ב-trigger או ב-service ביצירת/עדכון center_point
    -- שדות רווחיות (ליבת המערכת)
    quoted_price        NUMERIC(10,2),    -- מחיר ההצעה ₪
    actual_hours        NUMERIC(6,2),     -- שעות עבודה בפועל
    hourly_cost         NUMERIC(8,2),     -- עלות שעת עבודה ₪
    -- profit, profit_margin_pct, effective_hourly_rate מחושבים כ-GENERATED COLUMNS
    profit              NUMERIC(10,2) GENERATED ALWAYS AS
                        (quoted_price - (actual_hours * hourly_cost)) STORED,
    profit_margin_pct   NUMERIC(5,2) GENERATED ALWAYS AS
                        (CASE WHEN quoted_price > 0
                              THEN ((quoted_price - (actual_hours * hourly_cost)) / quoted_price * 100)
                              ELSE NULL END) STORED,
    source_dwg_path     VARCHAR(1000),
    source_folder       VARCHAR(1000),
    survey_boundary     GEOMETRY(Polygon, 2039),
    center_point        GEOMETRY(Point, 2039),
    bbox                GEOMETRY(Polygon, 2039),
    -- שדות עתידיים לחיזוי
    complexity_score    NUMERIC(4,2),
    revision_count      INTEGER DEFAULT 0,
    notes               TEXT,
    created_at          TIMESTAMPTZ DEFAULT NOW(),
    updated_at          TIMESTAMPTZ DEFAULT NOW()
);

-- אינדקסים מרחביים
CREATE INDEX idx_projects_boundary ON projects USING GIST (survey_boundary);
CREATE INDEX idx_projects_center ON projects USING GIST (center_point);
CREATE INDEX idx_projects_gush_helka ON projects (gush, helka);
CREATE INDEX idx_projects_date ON projects (survey_date);
CREATE INDEX idx_projects_profit ON projects (profit_margin_pct);  -- לדשבורד הרווחיות
```

## 🤖 שכבת אוטומציה וחיזוי

המערכת כוללת **שתי משפחות** של תהליכים אוטומטיים: אוטומציה דטרמיניסטית (פועלת מהיום הראשון) וחיזוי מבוסס-נתונים (פועל אחרי שיש מסה קריטית של פרויקטים מתויגים).

### 🔄 אוטומציה דטרמיניסטית (משלב 1)

#### 1. Watcher לתיקיות עבודה
תהליך רקע (`watchdog`) שמזהה DWG חדשים בתיקיות מוגדרות מראש. כשנמצא קובץ חדש:
- מריץ אוטומטית את ODA → DXF → חילוץ פוליגון
- יוצר רשומת `project` עם מטא-דאטה אוטומטית (תאריך, נתיב)
- שולח התראה לדפדפן ("נמצא DWG חדש, השלם פרטים")

#### 2. Spatial Join לגושים וחלקות
כשנוצר פרויקט חדש עם פוליגון, מתבצע אוטומטית spatial join לשכבת הגושים והחלקות, ומילוי השדות `gush` ו-`helka` אם אינם ידועים.

#### 3. בדיקת חפיפה עם מדידות עבר
לפני שהמשתמש סוגר רשומה חדשה, המערכת מציגה אוטומטית: "🔔 באזור הזה בוצעו 3 מדידות בעבר (2019, 2021, 2024). האם לראות?"

#### 4. ניקוי ועקביות יומית
cron יומי שבודק:
- פרויקטים בסטטוס "הושלם" בלי תאריך סיום
- פוליגונים מחוץ לגבולות ישראל
- שדות חובה ריקים

תוצאה: התראה במסך הדשבורד עם רשימת בעיות לתיקון.

#### 5. דוחות תקופתיים אוטומטיים
בכל סוף חודש, המערכת מייצרת PDF עם סטטוס המשרד ושומרת אותו בתיקיית "דוחות חודשיים". אין צורך בפעולה של המשתמש.

### 🎯 חיזוי רווחיות מבוסס כללי-אצבע (משלב 1 — מהיום הראשון)

כדי שהמערכת תוכל לחזות רווחיות **לפני** שיש מסה קריטית של נתונים, מוטמע מודל היוריסטי שמשתמש בכללי אצבע מקצועיים. כל המקדמים ניתנים לעריכה דרך מסך הגדרות, ומתעדכנים עם הניסיון.

#### Features שנכנסים לחיזוי

| מקור | Feature | חישוב | יחידות |
|-------|---------|--------|---------|
| 🤖 אוטומטי מ-DWG | `area_sqm` | שטח הפוליגון | מ"ר |
| 🤖 אוטומטי מ-DWG | `shape_index` | מדד צורה איזופרימטרי: `perimeter² / (4π × area)` | חסר יחידות |
| 🤖 אוטומטי מהמפה | `distance_from_office_km` | מרחק מ-`center_point` של הפרויקט למיקום המשרד (`ST_Distance` ב-PostGIS) | ק"מ |
| 👤 מהמשתמש | `survey_type` | סוג המדידה (רישום / טופוגרפית / להיתר וכו') | קטגורי |
| 👤 מהמשתמש | `site_type` | תנאי השטח: פנוי / מעורב / בנוי | קטגורי |

> 💡 **למה `shape_index` ולא יחס היקף/שטח?** היחס הפשוט P/A תלוי בגודל החלקה (ריבוע קטן וריבוע גדול מקבלים ערכים שונים למרות שזו אותה צורה). `shape_index` נורמלי לגודל ומדבר רק על **מורכבות צורה**: עיגול = 1.0, ריבוע = 1.27, רצועה ארוכה וצרה = 2.0+. כך אין כפילות בין `area` (שמייצג גודל) לבין `shape_index` (שמייצג צורה בלבד).

> 💡 מיקום המשרד נשמר בטבלת הגדרות (`settings.office_location` — נקודת PostGIS). אם המשרד עובר מקום, רק הערך הזה משתנה ו-`distance_from_office_km` מחושב מחדש לכל הפרויקטים.

> 💡 `perimeter_to_area_ratio` משקף עלות אמיתית: חלקה קטנה או צורה מפותלת = יותר עבודת גבול ביחס לשטח. לחלקה עגולה גדולה היחס נמוך; לחלקה מרובעת קטנה או חלקה ארוכה וצרה היחס גבוה.

#### נוסחת החיזוי

```
predicted_hours = (base_hours × shape_multiplier × site_multiplier) + travel_hours
```

**1. שעות בסיס (`base_hours`)** — לפי סוג המדידה וגודל:

```
base_hours = area_sqm × hours_per_sqm[survey_type]
```

| `survey_type` | `hours_per_sqm` (ברירת מחדל — עריך) |
|----------------|--------------------------------------|
| מדידה טופוגרפית | 0.002 |
| מדידה לתכנון | 0.0015 |
| מדידה לקדסטר | 0.001 |

**2. מקדם צורה (`shape_multiplier`)** — קונס לפי מדד הצורה האיזופרימטרי, **לא תלוי בגודל**:

```
shape_multiplier = 1.0 + (shape_index − ref_shape) × shape_sensitivity
ref_shape (ברירת מחדל): 1.27 (ריבוע — הצורה הנפוצה ביותר במגרשים)
shape_sensitivity (ברירת מחדל): 0.50
clipped to range [0.85, 1.50]
```

| צורה | `shape_index` | `shape_multiplier` |
|-------|---------------|---------------------|
| עיגול מושלם | 1.00 | 0.87 |
| ריבוע | 1.27 | 1.00 |
| מלבן 2:1 | ~1.43 | 1.08 |
| מלבן 4:1 | ~1.99 | 1.36 |
| רצועה ארוכה וצרה 10:1 | ~3.85 | 1.50 *(clipped)* |

**3. מקדם תנאי שטח (`site_multiplier`)**:

| `site_type` | מקדם |
|--------------|------|
| `open` (שטח פנוי) | ×1.00 |
| `mixed` (מעורב) | ×1.25 |
| `built` (שטח בנוי) | ×1.60 |

**4. שעות נסיעה (`travel_hours`)** — נוספות לסוף החישוב (לא מכפיל, כי הן עלות קבועה לפרויקט שלא משתנה עם גודלו):

```
travel_hours = (distance_from_office_km / avg_speed_kmh) × 2
```

| פרמטר | ברירת מחדל (עריך) | הסבר |
|--------|--------------------|--------|
| `avg_speed_kmh` | 60 | מהירות נסיעה ממוצעת (קמ"ש) |
| `2` | קבוע | הלוך + חזור |

**דוגמה לחישוב נסיעה**: פרויקט במרחק 25 ק"מ:
```
travel_hours = (25 / 60) × 2 = 0.83 שעות
```

> 💡 הנוסחה מניחה ביקור אחד באתר. אם פרויקט בפועל דרש כמה ביקורים — זה ישתקף ב-`actual_hours` שיוזן בסגירה, וכך מנגנון הכיול של שלב 4 יילמד מהסטייה.

#### חיזוי הרווח לפנייה חדשה

```
predicted_profit_at_price(P) = P − (predicted_hours × hourly_cost)
```

כשנכנסת פנייה חדשה, המסך מציג:
> *"לחלקה הזו (1,200 מ"ר, מדד צורה 1.35 — מלבן מאורך, שטח בנוי, מדידה לקדסטר, 25 ק"מ מהמשרד): צפי שעות עבודה + שעות נסיעה. עלות פנימית מחושבת. להצעת מחיר נתונה → רווח צפוי + שולי רווח באחוזים"*

> ⚠️ כל המקדמים לעיל הם **ברירת מחדל ראשונית**. הם נשמרים בטבלת `prediction_rules` ב-DB, וניתנים לעריכה בכל רגע דרך מסך הגדרות. המטרה: שהמערכת תעבוד מהיום הראשון, ושהמקדמים יתעדנו עם הזמן.

### 🎯 חיזוי מתקדם מבוסס-נתונים (משלב 4)

> ⚠️ **חוק זהב**: יכולת חיזוי מבוססת-נתונים מחליפה את ההיוריסטיקה **רק** אחרי שיש לפחות 100 פרויקטים סגורים עם `quoted_price` + `actual_hours` ממולאים. עם פחות מזה, המודל מייצר רעש סטטיסטי שמתחפש לתובנה. עד אז — היוריסטיקה.

#### 1. מנוע "פרויקטים דומים" — k-Nearest Neighbors
אלגוריתם KNN שמוצא 5 פרויקטים דומים ביותר לפנייה חדשה, לפי:
- קרבה גיאוגרפית (35%)
- `site_type` תואם (20%)
- `survey_type` תואם (20%)
- דמיון בגודל ויחס היקף/שטח (15%)
- סוג לקוח (10%)

#### 2. כיול מקדמי ההיוריסטיקה
ניתוח רגרסיה תקופתי שמשווה `predicted_hours` מול `actual_hours`, ומציע עדכון לטבלת `prediction_rules`. המנהל מאשר/דוחה כל שינוי.

#### 3. דיווח שגיאות חיזוי
המערכת מציגה את חמשת הפרויקטים האחרונים עם הסטייה הגדולה ביותר בין צפי לפועל — כדי לזהות איזה סוג עבודות תמיד "מתפוצצות" ולתקן את הכללים.

#### 2. זיהוי דפוסים עונתיים וגיאוגרפיים
ניתוח סדרות עת בסיסי שמזהה: "באזור X יש שיא ביקוש באביב", "מדידות להיתר גדלות ברבעון 4". פלט: גרפים בדשבורד.

#### 3. ניבוי עומס עבודה
דשבורד שמציג עומס מוערך לחודש הקרוב על סמך פרויקטים פעילים. כשנכנסת פנייה חדשה, המערכת מציגה: "אם תקבל את העבודה, שבוע 23 יעמיס ל-130%".

#### 4. ניבוי מורכבות (אבולוציוני)
מודל רגרסיה פשוט שצופה שעות עבודה לפרויקט חדש על סמך מאפיינים. נבנה רק אם המנוע ה-#1 מצליח ומשתמשים בו ביום-יום.

> 💡 **מה לא נכלל**: דירוג לקוחות, חיזוי "סיכוי לעבודה חוזרת", או כל ניתוח שעלול ליצור אפליה עסקית. דברים אלה נדחים ונשקלים מחדש רק אם יש בקשה מפורשת.

## 🔄 תהליכי עבודה — מחזור החיים המלא של פרויקט

המערכת בנויה סביב **לולאה אחת**: פנייה → חיזוי → תמחור מודע → ביצוע → סגירה → כיול → חיזוי טוב יותר לפנייה הבאה.

```
┌─ 1. פנייה חדשה ─┐    ┌─ 2. חיזוי אוטומטי ─┐    ┌─ 3. החלטת תמחור ─┐
│ מיקום           │ ──▶│ predicted_hours    │ ──▶│ סליידר מחיר      │
│ סוג מדידה       │    │ predicted_cost     │    │ → רווח חי         │
│ סוג שטח         │    │                    │    │                  │
└─────────────────┘    └────────────────────┘    └──────────────────┘
                                                          │
                                                          ▼
┌─ 7. כיול אוטומטי ─┐  ┌─ 6. סגירה ──────┐  ┌─ 5. ביצוע ─────┐  ┌─ 4. אישור ──┐
│ רגרסיה על סטיות   │◀─│ actual_hours    │◀─│ DWG מתעדכן    │◀─│ in_progress │
│ עדכון מקדמים      │  │ profit מחושב    │  │ Watcher רץ    │  │             │
└───────────────────┘  └─────────────────┘  └────────────────┘  └─────────────┘
```

### שלב 1 — פנייה חדשה נכנסת

**מי**: המנהל מקבל פנייה (טלפון, מייל, פגישה).

**מה הוא מזין במסך "פנייה חדשה"** (3-4 שדות):
- מיקום: כתובת / גוש-חלקה / סימון על המפה
- `survey_type`: טופוגרפית / לתכנון / לקדסטר
- `site_type`: פנוי / מעורב / בנוי
- (אופציונלי) DWG/DXF אם הלקוח שלח גאומטריה

---

### שלב 2 — חיזוי אוטומטי (מיידי)

המערכת מחשבת לבדה:

| Feature | מקור |
|---------|------|
| `area_sqm`, `perimeter_m`, `shape_index` | מהפוליגון (DWG אמיתי או סימון משוער על המפה) |
| `distance_from_office_km` | `ST_Distance` בין `center_point` למיקום המשרד |

ומריצה את הנוסחה:

```
predicted_hours = (area × hours_per_sqm[survey_type] × shape_mult × site_mult)
                + (distance_km / 60) × 2
```

```
┌──────────────────────────────────────────┐
│  צפי שעות עבודה:    8.3 ש'                │

│  ────────────────────────                 │
│  סה"כ שעות:         9.1 ש'                │

│                                          │
│  ╔════════════════════════════════════╗  │
│  ║  מחיר הצעה:  [── סליידר ──] ₪4,500 ║  │
│  ╚════════════════════════════════════╝  │
│                                          │
│  → רווח צפוי:       ₪2,680                │

```

הסליידר מעדכן את הרווח **בזמן אמת**. המנהל מקבל החלטה מודעת לתמחור — לא תחושה.

### שלב 4 — אישור פרויקט

הפנייה אושרה ע"י הלקוח → סטטוס `in_progress`. ב-DB נשמרים:
- `quoted_price` (מה שסגרנו)
- `predicted_hours` (החיזוי המקורי — שמור ל-Hindsight)
- כל ה-features ששימשו לחיזוי

---

### שלב 5 — ביצוע הפרויקט (ברקע)

המודד עובד בשטח ושומר DWG בתיקיית הפרויקט.

**Watcher אוטומטי**:
1. ODA Converter ממיר DWG → DXF
2. ezdxf מחלץ פוליגון משכבה M1200
3. המערכת מעדכנת `area_sqm`, `perimeter_m`, `shape_index` בערכים האמיתיים
4. Spatial join לגושים/חלקות
5. בדיקה: אם הגאומטריה האמיתית סוטה מהמשוערת ב-20%+ → **התראה למנהל** לעדכן חיזוי

---

### שלב 6 — סגירת פרויקט (הרגע הקריטי)

המנהל ממלא חובה:
- `actual_hours` — סך השעות שהושקעו בפועל
- `hourly_cost` — עלות שעת עבודה (יורש מ-default גלובלי)
- סטטוס → `completed`

ה-DB מחשב אוטומטית (GENERATED COLUMNS):
```
profit = quoted_price − (actual_hours × hourly_cost)
profit_margin_pct = profit / quoted_price × 100
```

הפרויקט הופך לנקודת **Hindsight** + דאטה אימון ל-**Foresight** עתידי.

---

### שלב 7 — לולאת כיול (אוטומטית, ברקע)

עם כל פרויקט שנסגר נשמרת הסטייה:
```
prediction_error = (actual_hours − predicted_hours) / predicted_hours
```

**כשמצטברים 100 פרויקטים סגורים** (שלב 4 ב-Roadmap):
- רגרסיה אוטומטית על הסטיות → המערכת מציעה עדכון למקדמי `prediction_rules`
- הודעה למנהל: *"לפי 23 פרויקטים אחרונים, `hours_per_sqm[קדסטר]` כדאי לעלות מ-0.001 ל-0.0013. לאשר?"*
- מודל KNN רץ במקביל להיוריסטיקה — מציג טווחי שעות לפי פרויקטים דומים בפועל

---

### תצוגות זמינות תמיד (לא חלק מהלולאה, אבל נסמכות עליה)

| מסך | מה הוא עונה |
|------|--------------|
| 🗺️ **מפת רווחיות** | איפה הרווחנו, איפה הפסדנו (פוליגונים צבועים לפי `profit_margin_pct`) |
| 📊 **דשבורד Hindsight** | התפלגות רווח לפי `survey_type`, `site_type`, מרחק, גודל |
| 🔍 **חיפוש מרחבי** | קליק על אזור → פרויקטים שם, רווחיותם, וקישור לתיקיות |
| 📅 **דוח חודשי אוטומטי** | סיכום שולי רווח של החודש, השוואה לחודשים קודמים |
