## 🔭 סקירה כללית

**SurveyGIS** היא מערכת מקומית לניהול וניתוח של מדידות במשרד מדידות. המערכת מאפשרת לקלוט קבצי DWG ישירות מתיקיות העבודה, לחלץ מהם פוליגוני תחום מדידה ומטא-דאטה, לשמור הכל בבסיס נתונים מרחבי, ולחקור את המידע על מפה אינטראקטיבית.

המערכת מתוכננת כ**כלי ייעודי וצנוע**, לא כפלטפורמת GIS מלאה. המטרה היא לפשט ולמקד — להציג רק את היכולות שמשרד מדידות באמת צריך ליום-יום, במקום להציע מאות אפשרויות שמרביתן אינן רלוונטיות.

### 🌐 למה שרת מקומי בדפדפן?

| סיבה | הסבר |
|-------|-------|
| 🔒 **פרטיות מלאה** | קבצי המדידה לא עוזבים את המחשב — לא לענן, לא לשירות צד שלישי |
| ⚡ **גישה לכלי דסקטופ** | המרת DWG דורשת תוכנה מקומית (ODA File Converter), והשרת המקומי קורא לה ישירות |
| 🌍 **חוויית Web מלאה** | פיתוח באמצעות טכנולוגיות web סטנדרטיות (HTML/CSS/JS + Leaflet) עם כל היתרונות, בלי תלות באינטרנט |
| 💰 **עלות אפס** | אין שרת בענן, אין מנויים, אין הוצאות חודשיות |
| 🛠️ **פשטות תחזוקה** | רכיב אחד פיזי לתחזוקה — המחשב של מנהל המשרד |

### מה המערכת מאפשרת

- **קליטה אוטומטית של DWG** מתיקיות עבודה — חילוץ פוליגון תחום המדידה ומטא-דאטה
- **תיוג והעשרה ידנית** של מטא-דאטה עסקית (לקוח, סוג מדידה, גוש/חלקה וכו')
- **חיפוש מרחבי** — לחיצה על מיקום במפה ושליפת כל המדידות באזור
- **סינון מתקדם** — לפי תאריכים, סוג מדידה, לקוח, סטטוס, וגודל
- **תצוגה מרחבית** של פיזור המדידות עם heatmaps וקלאסטרינג
- **דשבורד תפעולי** — עומסי עבודה, מגמות, ופרויקטים פעילים
- **מנוע "פרויקטים דומים"** — איתור מדידות עבר רלוונטיות לכל פנייה חדשה
- **יצוא נתונים** ל-CSV, GeoJSON, KMZ, ו-PDF

## 🧭 פילוסופיית התכנון

המערכת בנויה סביב חמישה עקרונות שמכוונים כל החלטה:

**1. פשטות לפני שלמות** — עדיף לעשות מעט דברים מצוין מאשר להציע הכל באופן בינוני. כל פיצ'ר שלא משרת שאלה עסקית קונקרטית — נזרק.

**2. למידה לפני בנייה** — לפני שבונים GUI ייעודי, משתמשים ב-QGIS כקרקע אימונים כדי להבין מה באמת נחוץ. ה-GUI מבוסס על שימוש בפועל, לא על השערות.

**3. הגבלה כתכונה** — פלטפורמות GIS אוניברסליות מציעות מאות אפשרויות שלא רלוונטיות לעבודת משרד מדידות. הכלי הייעודי מציע מסך אחד עם 5 כפתורים — וזה מה שהופך אותו ליעיל יותר.

**4. נתונים לפני אלגוריתמים** — לפני שמדברים על ML וחיזוי, צריך מאגר נתונים נקי ומתויג. הסכמה כוללת מהיום שדות עתידיים (גם אם ריקים), כך שהמערכת מוכנה לאינטליגנציה מתי שתגיע.

**5. ROI מובהק לכל יכולת** — כל פיצ'ר חיזוי או אוטומציה חייב לעבור מבחן "כמה שעות זה חוסך לי בחודש?". אם אין תשובה ברורה — לא בונים.

## 👥 משתמשי המערכת

המערכת מיועדת ל-**1–2 משתמשים** מנהלי משרד מדידות. **אין** הפרדת הרשאות, **אין** ניהול משתמשים, **אין** רישום וכניסה. המערכת מניחה משתמש מורשה אחד שמפעיל את השרת המקומי על המחשב שלו.

### תרחישי שימוש מרכזיים

| תרחיש | תיאור |
|--------|--------|
| 📞 **פנייה חדשה מלקוח** | המנהל מקבל פנייה על חלקה. תוך 15 שניות הוא רואה במפה אם עבד באזור, איזה פרויקטים דומים בוצעו, וכמה הם עלו |
| 📥 **קליטת מדידה חדשה** | מודד שומר DWG בתיקיית העבודה. המערכת מזהה את הקובץ, מחלצת את הפוליגון, ומבקשת מהמנהל למלא את המטא-דאטה |
| 🔍 **חיפוש מדידה ישנה** | המנהל מחפש לפי גוש/חלקה, שם לקוח, או תאריך, ומקבל קישור ישיר לתיקיית הפרויקט במחשב |
| 📊 **סקירה חודשית** | בכל סוף חודש, המנהל פותח דשבורד שמציג כמה פרויקטים נסגרו, איפה הם בוצעו, ומה הצפי לחודש הבא |
| 💼 **תמחור פרויקט** | בהכנת הצעת מחיר, המנהל רואה את העלויות וזמני הביצוע של 5 פרויקטים דומים בעבר |

## 📥 קלט — סוגי קבצים ומקורות מידע

### קבצי מדידה נתמכים

| פורמט | סיומת | תיאור | תהליך |
|--------|--------|--------|--------|
| AutoCAD DWG | `.dwg` | פורמט מקורי של AutoCAD / Civil3D | ODA File Converter ממיר ל-DXF → ezdxf מחלץ |
| AutoCAD DXF | `.dxf` | פורמט חליפין | ezdxf קורא ישירות |
| CSV | `.csv` | טבלאות קואורדינטות (X, Y, Z) | pandas + shapely → בניית גאומטריה |
| GeoJSON | `.geojson` | פורמט גאוגרפי מבוסס JSON | geopandas קורא ישירות |
| Shapefile | `.shp` + נלווים | קבצי GIS סטנדרטיים | geopandas/fiona קורא ישירות |

> ⚠️ **מודולים נוספים (KML, GPX, LAS, תמונות אווריות) נדחים לשלבים מאוחרים** — לא נכללים ב-MVP. נוספים רק במידה ויש צורך אמיתי שעולה משימוש בפועל.

### מה מחלצים מקובץ DWG/DXF

| ישות | מהי | למה |
|-------|------|------|
| 🟦 **פוליגון תחום מדידה** | LWPOLYLINE/POLYLINE סגור מהשכבה המוסכמת (M1200) | ה"מי" העיקרי של המדידה — האזור שעליו עבדנו |
| 🟢 **נקודות מדידה** (אופציונלי) | POINT / INSERT (בלוקים) מהשכבות הרלוונטיות | לטווח גובה ומאפייני שטח |
| 📐 **מערכת קואורדינטות** | ממטא-דאטה של הקובץ או הגדרת ברירת מחדל | לאחידות והמרות |
| 📅 **תאריך עריכת הקובץ** | מ-file system metadata | כברירת מחדל לתאריך מדידה |
| 📁 **נתיב מקור** | התיקייה ושם הקובץ במחשב | לקישור חזרה לקובץ המקורי |

### מטא-דאטה ידנית (מוזנת על ידי המשתמש)

| שדה | סוג | חובה | תיאור |
|------|-----|------|--------|
| `project_name` | טקסט | ✅ | שם תיאורי לפרויקט |
| `survey_type` | בחירה מרשימה | ✅ | רישום / טופוגרפית / להיתר / לפיצול / תשתיות / תכנון |
| `client_name` | טקסט | ✅ | שם הלקוח / חברה |
| `survey_date` | תאריך | ✅ | תאריך ביצוע המדידה בשטח |
| `gush` | מספר | ⚪ | מספר גוש |
| `helka` | מספר | ⚪ | מספר חלקה |
| `address` | טקסט | ⚪ | כתובת או אזור |
| `city` | טקסט | ⚪ | עיר / יישוב |
| `status` | בחירה | ✅ | בעבודה / הושלם / הוגש / ארכיון |
| `quoted_price` | מספר | ⚪ | מחיר הצעת המחיר |
| `actual_hours` | מספר | ⚪ | שעות עבודה בפועל |
| `surveyor_notes` | טקסט חופשי | ⚪ | הערות פנימיות |

> 💡 **שדות עתידיים** — הסכמה כוללת גם שדות שלא ימולאו ב-MVP אבל ישמשו לחיזוי מאוחר יותר: `complexity_score`, `revision_count`, `client_repeat_count`. נשמרים ב-DB מהיום כדי לא לשבור תאימות אחורה.

### מקורות מידע חיצוניים

| מקור | שימוש | שלב |

## 🗄️ בסיס נתונים

### טכנולוגיה

**PostgreSQL 16 + PostGIS 3.4** — מקומי, רץ על אותו מחשב כמו השרת.

> 💡 **למה לא SQLite + SpatiaLite?** — SQLite היה מספיק לחלוטין ל-1–2 משתמשים, אבל PostGIS נותן יכולות מרחביות עשירות יותר (בעיקר אינדקסים GIST, ופונקציות ניתוח מתקדמות) שיהיו חשובות בשלב 4 לחיזוי וניתוח. ההתקנה חד-פעמית, התחזוקה אפסית.

## 🤖 שכבת אוטומציה וחיזוי

המערכת כוללת **שתי משפחות** של תהליכים אוטומטיים: אוטומציה דטרמיניסטית (פועלת מהיום הראשון) וחיזוי מבוסס-נתונים (פועל אחרי שיש מסה קריטית של פרויקטים מתויגים).

### 🔄 אוטומציה דטרמיניסטית (משלב 1)

#### 1. Watcher לתיקיות עבודה
תהליך רקע (`watchdog`) שמזהה DWG חדשים בתיקיות מוגדרות מראש. כשנמצא קובץ חדש:
- מריץ אוטומטית את ODA → DXF → חילוץ פוליגון
- יוצר רשומת `project` עם מטא-דאטה אוטומטית (תאריך, נתיב)
- שולח התראה לדפדפן ("נמצא DWG חדש, השלם פרטים")

#### 2. Spatial Join לגושים וחלקות
כשנוצר פרויקט חדש עם פוליגון, מתבצע אוטומטית spatial join לשכבת הגושים והחלקות, ומילוי השדות `gush` ו-`helka` אם אינם ידועים.

#### 3. בדיקת חפיפה עם מדידות עבר
לפני שהמשתמש סוגר רשומה חדשה, המערכת מציגה אוטומטית: "🔔 באזור הזה בוצעו 3 מדידות בעבר (2019, 2021, 2024). האם לראות?"

#### 4. ניקוי ועקביות יומית
cron יומי שבודק:
- פרויקטים בסטטוס "הושלם" בלי תאריך סיום
- פוליגונים מחוץ לגבולות ישראל
- שדות חובה ריקים

תוצאה: התראה במסך הדשבורד עם רשימת בעיות לתיקון.

#### 5. דוחות תקופתיים אוטומטיים
בכל סוף חודש, המערכת מייצרת PDF עם סטטוס המשרד ושומרת אותו בתיקיית "דוחות חודשיים". אין צורך בפעולה של המשתמש.

### 🎯 חיזוי מבוסס-נתונים (משלב 4)

> ⚠️ **חוק זהב**: כל יכולת חיזוי נבנית **רק** אחרי שיש לפחות 100 פרויקטים מתויגים נקי. עם פחות מזה, המודל מייצר רעש סטטיסטי שמתחפש לתובנה.

#### 1. מנוע "פרויקטים דומים" (הראשון להיבנות)
אלגוריתם k-Nearest Neighbors שמוצא 5 פרויקטים דומים ביותר לפנייה חדשה, לפי:
- קרבה גיאוגרפית (40%)
- סוג מדידה (25%)
- גודל פוליגון (20%)
- סוג לקוח (15%)

**ערך**: תמחור מבוסס נתונים. כשמגיעה פנייה, המנהל רואה מיד: "3 מדידות דומות בעבר עלו 8,000–12,000 ₪ ולקחו 4–7 ימים".

#### 2. זיהוי דפוסים עונתיים וגיאוגרפיים
ניתוח סדרות עת בסיסי שמזהה: "באזור X יש שיא ביקוש באביב", "מדידות להיתר גדלות ברבעון 4". פלט: גרפים בדשבורד.

#### 3. ניבוי עומס עבודה
דשבורד שמציג עומס מוערך לחודש הקרוב על סמך פרויקטים פעילים. כשנכנסת פנייה חדשה, המערכת מציגה: "אם תקבל את העבודה, שבוע 23 יעמיס ל-130%".

#### 4. ניבוי מורכבות (אבולוציוני)
מודל רגרסיה פשוט שצופה שעות עבודה לפרויקט חדש על סמך מאפיינים. נבנה רק אם המנוע ה-#1 מצליח ומשתמשים בו ביום-יום.

> 💡 **מה לא נכלל**: דירוג לקוחות, חיזוי "סיכוי לעבודה חוזרת", או כל ניתוח שעלול ליצור אפליה עסקית. דברים אלה נדחים ונשקלים מחדש רק אם יש בקשה מפורשת.

## 🗓️ Roadmap — תכנית פיתוח בארבעה שלבים

תכנית הפיתוח בנויה סביב **למידה לפני בנייה**. כל שלב מסתיים בפלט שמיש בפני עצמו, ומספק תובנות שמעצבות את השלב הבא.

### 🔵 שלב 1 — הגדרת שפת הנתונים (3 שבועות)

**מטרה**: סקריפט עובד שממיר DWG ל-CSV/GeoJSON עם מטא-דאטה מלאה, ומאגר ראשוני של 50–60 פרויקטים מתויגים.

**משימות**:
- [ ] סקריפט Python: DWG → DXF (ODA) → חילוץ פוליגונים → CSV/GeoJSON
- [ ] הגדרת קונבנציות שמות קבצים ומבנה תיקיות
- [ ] עיצוב סכמת DB מלאה (כולל שדות עתידיים)
- [ ] התקנת PostgreSQL + PostGIS מקומית
- [ ] כתיבת מיגרציות Alembic
- [ ] תיוג ידני של 50–60 מפות היסטוריות (מילוי מטא-דאטה ב-spreadsheet)
- [ ] טעינת הנתונים המתויגים ל-PostGIS

**פלט**: מאגר PostGIS עם 50+ פרויקטים, וסקריפט שעובד על כל DWG חדש.

---

### 🟢 שלב 2 — חקר באמצעות QGIS (חודשיים)

**מטרה**: ללמוד מה באמת חשוב לראות, איזה שאילתות חוזרות, ומה צריך להיכלל ב-GUI עתידי.

**משימות**:
- [ ] חיבור QGIS ל-PostGIS המקומי
- [ ] בניית פרויקט QGIS עם השכבות הרלוונטיות (פוליגונים, גושים, רקע)
- [ ] שימוש יומיומי ב-QGIS לעבודה אמיתית של המשרד (לפחות חודש)
- [ ] ניתוחים תיאוריים: heatmaps, צפיפות, התפלגות סוגי מדידה
- [ ] תיעוד שיטתי: יומן של "מה חיפשתי", "מה היה מסורבל", "מה חסר"
- [ ] כתיבת מסמך דרישות מסכם: 4–6 מסכים בלבד עם רשימת פיצ'רים מצומצמת

**פלט**: מסמך דרישות מבוסס-התנסות שמגדיר בדיוק מה ייבנה בשלב 3.

> ⚠️ **אל תקצר**: הפיתוי לדלג מהר משלב 2 לשלב 3 הוא הגדול ביותר בפרויקט. כל שבוע בשלב 2 חוסך 2 שבועות בשלב 3.

---

### 🟠 שלב 3 — בניית GUI ייעודי (3 חודשים)

**מטרה**: אפליקציית web מקומית עם GUI מותאם בדיוק לצרכי המשרד.

**משימות**:
- [ ] שרת Flask עם API פנימי
- [ ] Frontend עם Leaflet + Vanilla JS + Chart.js
- [ ] מסך 1: מפת סקירה עם פילטרים
- [ ] מסך 2: כרטיס פרויקט
- [ ] מסך 3: דשבורד תפעולי
- [ ] מסך 4: רשימת תוצאות חיפוש
- [ ] Watcher לקליטת DWG אוטומטית
- [ ] Spatial join אוטומטי לגושים/חלקות
- [ ] בדיקת חפיפה עם מדידות עבר
- [ ] יצוא בסיסי (CSV, GeoJSON, PDF)
- [ ] hook ראשוני ל"פרויקטים דומים" (קרבה גיאוגרפית בלבד)

**פלט**: מערכת מלאה בשימוש יומי, מחליפה את QGIS לרוב המשימות.

---

### 🟣 שלב 4 — אינטליגנציה אבולוציונית (פתוח, אחרי 6 חודשי שימוש)

**מטרה**: הוספת יכולות חיזוי מבוססות-נתונים — רק כשיש מסה קריטית של נתונים.

**תנאי כניסה**: לפחות 100 פרויקטים מתויגים נקי + 6 חודשי שימוש בשלב 3.

**משימות (לפי סדר עדיפויות)**:
- [ ] שדרוג מנוע "פרויקטים דומים" עם משקלות חכמים (KNN מלא)
- [ ] זיהוי דפוסים עונתיים — גרפים בדשבורד
- [ ] ניבוי עומס עבודה
- [ ] (אופציונלי) ניבוי מורכבות פרויקט
- [ ] (אופציונלי) heatmap דינמי של מגמות

**פלט**: מערכת שלא רק שומרת היסטוריה, אלא גם תומכת בקבלת החלטות עסקיות.

> 💡 כל יכולת בשלב הזה נבחנת בנפרד עם **מבחן ROI**: כמה שעות זה חוסך בחודש? כמה החלטות טובות יותר אני מקבל? אם התשובה "לא יודע" — לא בונים.
